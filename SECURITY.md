# セキュリティポリシー

`予定お知らせ君` のセキュリティ設計上の前提・制約・推奨をまとめます。

## 運用前提

このボットは **個人用・単一ユーザー** を前提に設計されています。`LINE_USER_ID` で指定した1人のLINEユーザーだけが使える想定です。

- 家族・知人・組織で共有する運用は想定外
- 公開ボット（誰でも友だち追加できる）としての運用は非推奨

公開運用や多人数運用へ拡張する場合は、本書末尾の「拡張時の注意」を参照してください。

---

## 実装しているセキュリティ対策

### 1. Script Properties による秘匿値管理

`LINE_ACCESS_TOKEN` や `LINE_USER_ID` はコードに直書きせず、Script Properties（スクリプトプロパティ）で管理します。

- コードを GitHub 等に公開してもトークンは漏れない
- コードの差し替えで秘匿値が飛ばない
- セットアップ手順は [README](README.md) 参照

### 2. LINE_USER_ID ホワイトリスト

`doPost` は Webhook イベントの `source.userId` が Script Properties の `LINE_USER_ID` と一致する場合のみ処理します。不一致なら早期 return します（`main.gs` の `handleWebhookEvent`）。

**効果**: GAS Web アプリ URL が漏洩しても、攻撃者の `userId` は `LINE_USER_ID` と一致しないため、偽装リクエストによるカレンダー改ざん等は発生しません。

**裏返し**: `LINE_USER_ID` を正しく設定しないと、自分からのメッセージもすべて拒否されます。

### 3. セッション状態管理 + LockService

タスク登録フローの各段階（開始時刻 → 終了時刻 → 確定）はセッション（Script Properties）で管理し、`LockService` で排他化しています。

**効果**:
- 古いボタンの再タップ → 現在のステップと不一致なら無視
- 同時連打 → ロック取得待ちでシリアル化、片方のみ登録

### 4. サーバー側バリデーション

`executeFinalRegistration` の冒頭で以下を検証します:
- タスク名の空文字・200文字超
- `Date` 解析失敗（`NaN`）
- 終了時刻が開始時刻以前

UI 側の入力ガードをすり抜けた異常値に対する防衛線です。

## 既知の制約

### 署名検証（`x-line-signature`）が未実装

**根本原因**: Google Apps Script の `doPost(e)` はHTTPリクエストヘッダにアクセスできません。LINE の署名は `x-line-signature` ヘッダで送られるため、純 GAS 構成では検証のしようがないという構造的制約があります。

**リスク範囲**: GAS Web アプリ URL（`https://script.google.com/macros/s/.../exec` の形式）は実質的にランダムな長い文字列で、知らなければ到達できない類の URL ではあります。ただし、もし URL が漏れると、偽の LINE イベントを POST される可能性が残ります。

**緩和策（現状）**:
- `LINE_USER_ID` ホワイトリストで「本人の userId と一致するリクエスト」のみ処理
- 攻撃者は本人の userId を知らない限り、カレンダー改ざん等は不能
- ただし、大量送信による GAS クォータ枯渇リスクは残る

**推奨運用**:
- Web アプリ URL を他人と共有しない・リポジトリ・ログに残さない
- 漏洩疑いがある場合は `デプロイ` から新しいデプロイを作成し URL を更新
- LINE チャネルアクセストークンも定期的に再発行すると尚良い

## 拡張時の注意

### 複数ユーザー運用にしたい場合

`LINE_USER_ID` を配列化してホワイトリスト判定を `.includes()` に変えるのは最小変更です。ただし:
- セッションキーは Script Properties で `task_session` 固定なので、ユーザーごとに `task_session:${userId}` で分離する必要がある
- 目標・タスクが1つのカレンダーに混在するので、ユーザーごとに別カレンダーを設定する設計変更も要検討

### 公開ボット（誰でも友だち追加）にしたい場合

署名検証の実装が必須になります。GAS 単独では不可能なので、Cloud Run / Cloud Functions / Vercel Functions 等の薄い中継層を置き、そこで `x-line-signature` を HMAC-SHA256 で検証してから GAS へ転送する構成に変更してください。

その際に追加で必要な対応:
- プライバシーポリシー・利用規約の明記（LINE 公式の要求）
- カレンダー分離（ユーザーごとに別カレンダー）
- レート制限
- 適切なエラー応答（個人情報を返さない）
